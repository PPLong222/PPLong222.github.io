缓存机制

RecyclerView缓存核心对象是什么： ViewHolder(包装了View)

RecyclerView缓存复用核心过程：当RecyclerView开始滑动时，有新的View进入，或老的View移出视角

RecyclerView四级缓存机制：

  • mAttachedScrap(屏幕内)：屏幕内itemView快速冲用，不需要重新createaView和bindView
  • mCachedViews(屏幕外)：最近才移除屏幕的VIE我H哦来的人，包括数据和position信息，复用时必须是相同位置的ViewHolder才能够复用，此时不用重新bindView
  • mViewCacheExtension(自定义缓存)
  • mRecyclerPool(缓存池)：数据和position等被清理，复用时需要重新bindView

由于RecyclerView流程较复杂，因此这里我们先从关键的方法中入手，既然要存在回收过程，那么我们先关注回收方法

如何进行复用？

这里肯定是要从触摸机制说起的，RecyclerView重写onTouchEvent对触摸事件进行监听，那么在ACTION_MOVE时肯定会调用滑动相关的方法， 这里因为自己时间原因，就先口述一下前置流程

  • 手指开始触摸RecyclerView且进行滑动，进入到ACTION_MOVE方法中
  • 层层调用，调用到其对应LayoutManager的scrollBy方法中，这里先以LinearLayoutManager为例
  • 在scrollBy方法中调用fill方法，进入到关键阶段

fill填充子控件

fill的主要核心是：先尝试回收RecyclerView中的View，然后看当前RecyclerView是否还有剩余空间可以填充，如果有的话通过layoutChunk方法进行填充，知道不能填充为止。

核心方法：layoutChunk，进行子控件的测量和布局

何时进行回收？

进入到我们之前所提到的fill方法中，当又填充元素到RecyclerView，此时可能需要对RecyclerView中其他元素进行回收，因此会调用到相关方法(同样，这里也暂时只看LinearLayoutManager).先总结一下流程

  • 在scroll或初始化时，会调用fill方法尝试填充元素
  • 当进行正常滑动时，或当填充完一个元素后，此时进行回收判断，根据具体LayoutState方向，选择合适的回收方法。
  • 判断该从哪个View开始回收，并将其之后不符合条件的View都依次调用回收方法(Recycler::recycleView)

// 主要看是该从哪个方向进行回收(看Layout布局是怎样的)
private void recycleByLayoutState(RecyclerView.Recycler recycler, LayoutState layoutState) {
    if (!layoutState.mRecycle || layoutState.mInfinite) {
        return;
    }
    int scrollingOffset = layoutState.mScrollingOffset;
    int noRecycleSpace = layoutState.mNoRecycleSpace;
    if (layoutState.mLayoutDirection == LayoutState.LAYOUT_START) {
        recycleViewsFromEnd(recycler, scrollingOffset, noRecycleSpace);
    } else {
      	// 调用recycleChildren方法
        recycleViewsFromStart(recycler, scrollingOffset, noRecycleSpace);
    }
}

private void recycleViewsFromStart(RecyclerView.Recycler recycler, int scrollingOffset, int noRecycleSpace) {
  final int childCount = getChildCount();
	final int limit = scrollingOffset - noRecycleSpace;
  // 找到待回收的View，开始回收从该View开始的剩下View(因为是from start)
  for (int i = 0; i < childCount; i++) {
        View child = getChildAt(i);
        if (mOrientationHelper.getDecoratedEnd(child) > limit
                || mOrientationHelper.getTransformedEndWithDecoration(child) > limit) {
            // stop here
            recycleChildren(recycler, 0, i);
            return;
        }
    }
}

private void recycleChildren(RecyclerView.Recycler recycler, int startIndex, int endIndex) {
    if (startIndex == endIndex) {
        return;
    }
    if (endIndex > startIndex) {
        for (int i = endIndex - 1; i >= startIndex; i--) {
            removeAndRecycleViewAt(i, recycler);
        }
    } else {
        for (int i = startIndex; i > endIndex; i--) {
            removeAndRecycleViewAt(i, recycler);
        }
    }
}


public void removeAndRecycleViewAt(int index, @NonNull Recycler recycler) {
    final View view = getChildAt(index);
  	// 将View从RecyclerView中移除
    removeViewAt(index);
  	// 关键方法
    recycler.recycleView(view);
}

如何进行回收？

mAttachedView回收

暂时还不是很清楚，似乎是在onLayout时进行复用，与滑动时无关

CachedView回收

此处是尝试对RecyclerView中的View进行回收，此时View已经不在主屏幕中了.在RecycleView的滑动过程中，回收的重点基本上是CacheView以及RecyclerPool中的ViewHolder

// RecyclerView.Recycler
public void recycleView(@NonNull View view) 
	// 尝试从View的LayoutParam中获取到ViewHolder(LayoutParam即RecycerlView.LayoutParam)
  // 此时的View应该是处于屏幕外的
	ViewHolder holder = getChildViewHolderInt(view);
	// 回收这个holder
	recycleViewHolderInternal(holder);
}

void recycleViewHolderInternal(ViewHolder holder) {
  if (forceRecycle || holder.isRecyclable()) {
		int cachedViewSize = mCachedViews.size();
    // 如果CachedView中存满了，就删除最老的那一个ViewHolder并存入到RecyclerPool中
    if (cachedViewSize >= mViewCacheMax && cachedViewSize > 0) {
        recycleCachedViewAt(0);
        cachedViewSize--;
    }
    int targetCacheIndex = cachedViewSize;
    // 将当前ViewHolder放入到mCachedView中
    mCachedViews.add(targetCacheIndex, holder);
    cached = true;
  }
  if (!cached) {
    addViewHolderToRecycledViewPool(holder, true);
    recycled = true;
  }
  // 将该holder从mViewInfoStore中删除
  mViewInfoStore.removeViewHolder(holder);
}

// 将该ViewHolder从CacheView中移除并加入到RecycledViewPoo中
void recycleCachedViewAt(int cachedViewIndex) {
    ViewHolder viewHolder = mCachedViews.get(cachedViewIndex);

    addViewHolderToRecycledViewPool(viewHolder, true);
    mCachedViews.remove(cachedViewIndex);
}

RecycledViewPool回收

那么当CacheView放满了后，ViewHolder是如何存入到RecyclerPool中的呢？

先了解一下RecycledViewPool结构

public static class RecycledViewPool {
  	// 单个类型的Scrap最大缓存数量
    private static final int DEFAULT_MAX_SCRAP = 5;
  	// 缓存指定ViewType的数据结构
    static class ScrapData {
      	// 缓存集合(实质是ArrayList<ViewHolder>)
        final ArrayList<ViewHolder> mScrapHeap = new ArrayList<>();
      	// 最大缓存数(默认为5)

        long mBindRunningAverageNs = 0;

}

// RecycledViewPool
public void putRecycledView(ViewHolder scrap) {
  	// 获得该ViewHolder 的ViewType。关键
    final int viewType = scrap.getItemViewType();
  	// 以viewType为key查找对应的ScrapData集合
    final ArrayList<ViewHolder> scrapHeap = getScrapDataForType(viewType).mScrapHeap;
  	// 如果该viewType缓存队列当前可缓存对象已经满了，则就不在再添加缓存了(因为缓存的都只是没有数据的View，都是一样的Type)
    if (mScrap.get(viewType).mMaxScrap <= scrapHeap.size()) {
      return;
    }
  	// 重置ViewHolder的基本属性(包括mPosition、mFlags、mItemId，还会清除其payload)
    scrap.resetInternal();
    scrapHeap.add(scrap);
}

总结一下，当RecyelerView确定要回收一个View时，那么它会做以下的步骤：

  1. 提取该View的ViewHolder，后续操作都是基于ViewHolder的
  2. 看能不能回收进CachedView中(最大容量为2，可能存在预取机制)
    1. 如果能，则直接放入CachedView中
    2. 如果CachedView放满了，则在缓存池中尝试通过其ViewType放入到对应的队列中(ScrapData，最大元素个数为5个).若缓存池都放满了，那就不管

预取机制

这最新这几个版本中，使用LinearLayoutManager默认开启了预取机制(Prefetch)。目前还没有细看代码，但功能差不多就是在view还没进入到视野之前就试图通过复用或创建该ViewHolder并缓存mCachedView(使得起默认最大值为3)，可以通过LinearLayoutManager::setItemPrefetchEnabled来设置相关参数
