Linux文件系统理解

fs是啥？文件系统？

一直不懂make rootfs fs是啥，Linux文件系统怎么存储，看了下博客抄了一些话

文件系统

文件系统的基本组成

文件系统是操作系统中负责管理持久数据的子系统，说简单点，就是负责把用户的文件存到磁盘硬件中，因为即使计算机断电了，磁盘里的数据并不会丢失，所以可以持久化的保存文件。
文件系统的基本数据单位是文件，它的目的是对磁盘上的文件进行组织管理，那组织的方式不同，就会形成不同的文件系统。
Linux 最经典的一句话是：「一切皆文件」，不仅普通的文件和目录，就连块设备、管道、socket 等，也都是统一交给文件系统管理的。

Linux文件系统回味每个文件分配两个数据结构： 索引节点和目录项，记录文件的原信息和目录层次结构

  • 索引节点：index node – inode 。记录文件的元信息、文件大小、访问权限、创建时间、修改时间、数据在磁盘的中的位置…是文件的唯一标识，一一对应，占用磁盘空间。为加速文件的访问，通常会把索引节点加载到内存中
  • 目录项： directory entry – dentry，记录文件的名字、索引节点指针和其他目录项的层级关系，多个目录项关联就会形成目录结构，但与索引节点不同，目录项是内核维护的一个数据结构，不存放在磁盘，而是在内存中

目录项和索引项的关系是多对一
目录也是文件，用索引节点唯一表示，但在磁盘里保存子目录或者文件；而普通文件在磁盘离保存的是文件数据

目录项 != 目录

目录是文件，持久化在磁盘中，目录项是内核的数据结构，缓存在内存中
查询目录频繁从磁盘读 ->效率低->内核把已经度过的目录用目录项数据结构缓存在内存中，提高fs的效率

文件数据如何存在磁盘中？

磁盘读最小单位是扇区 —512B ，如果每次都写这么小的单位，则效率很低
所以fs把多个扇区组成一个逻辑快，每次读写的最小单位就是逻辑块（数据块）。Linux中逻辑块大小为4KB,提高了磁盘读写效率

磁盘格式化时，分成三个存储区域：

  • 超级块，存储fs的详细信息，块个数、块大小、空闲块。fs挂载时加载到内存
  • 索引节点区：存储索引节点。文件被访问时加载到内存中
  • 数据块区：存储文件或者目录数据

VFS

前提：fs类型 多，os希望对用户提供一个统一的接口，一个代理人。“没有什么问题是加一层解决不了的”。于是用户层与os层之间就加了一层，叫虚拟文件系统 VFS

VFS定义了一组所有fs都支持的数据结构和标准接口，程序员不需要了解所欲文件系统的工作原理，只需要知道VFS提供的API就行

又想到了今天读内核的设计和实现中的话

“提供机制，而不是策略。”

根据存储位置不同，fs可分为三类：

  • 磁盘的fs，数据直接存储在磁盘中 例如Ext2/3/4、XFS
  • 内存的fs，数据不存在磁盘，而是在内存之上 /proc 和/sys的fs都属于这一类，读写这类文件，实际上是读写内核中先关的数据
  • 网络fs，用来访问其他计算机主机数据的文件系统，比如NFS、SMB等

fs首先要挂在到某个目录才能正常使用，比如Linux在启动时会把系统文件挂在到根目录

文件描述符

感觉有点像pcb/task_struct一样的东西

os会跟踪进程打开的所有文件，os位每个进程维护一个打开文件表，文件表离的每一项都是文件描述符

$$进程 \xrightarrow{1 \space \space n}文件表\xrightarrow{1 \space \space n}文件描述符 $$

这里我还没学到Linux内核层面的文件描述符，估计以后会看到，先插个眼

包含文件指针、文件打开计数器、文件磁盘位置、访问权限…

fs基本操作单位是数据块

文件的存储方式

连续空间存放

文件的数据连续发存放在磁盘中，紧密相联，读取数据效率高（与因为一次磁盘寻道就可以读出整个文件）

前提：必须知道一个文件的大小，这样fs才会根据文件的阿晓在磁盘上找到一块连续的空间分配给文件。文件头需要指定起始块位置和长度。

但是，文件如何扩容？？？

问题：导致磁盘空间碎片和文件长度不易扩展（有JVM垃圾清理内味了）

非连续空间存放

  1. 链表方式

    离散的、不连续的 -> 消除了磁盘碎片且文件长度可动态扩展

    1. 隐式链表

      文件头包含第一块和最后一块的位置，每个数据块都有一个指针指向下一块的位置（合理）

      问题：

      • 无法直接访问数据块，只能通过指针顺序访问，不能根据索引直接访问（链表本身就存在的问题，如果你只向读一个文件的一点内容，那必须要把它之前的内容都过一遍）
      • 数据块指针消耗了一定的空间分配id稳定性交叉，可能导致链表中的指针丢失而导致文件数据丢失(???)

    2. 显示链接

      把用于连接文件各个数据块的指针，显示的存放在内存的一张链表（文件分配表 – File Allocation Table FAT）里，该表在整个磁盘中仅设置一张，每个表项存放连接指针，指向下一个数据块号（磁盘中这么多文件啊？是否数据量过大？还是说是动态加载类型？）

      不太懂是怎么存的，每一个块都要在表中有记录？这种方式下，一个数据块只能分配给一个文件吗，感觉不合理啊？？？

      （答：破案了家人们，一个逻辑块只能存放一个文件内容，无论这个有多小，暂时对ext2来说是的，可以通过stat filename来查看）

      特点：

      • 查找过程在内存中进行，速度快，减少了磁盘IO
      • 缺点：不适用于大磁盘—> 基于内存

  2. 索引方式

    链表不能直接有效的访问

    为每个文件创建了一个索引数据块，里面存放的是指向文件数据块的列表指针

    文件头需要包含指向索引数据块的指针

    • 文件的创建、增缩方便
    • 不会有碎片问题
    • 支持顺序读写和随机读写

    问题：

    • 存在磁盘中，额外的开销

    如果一个文件很大，以至于一个数据块放不下索引信息，那要如何处理大文件的存放？ — 组合方式

    链式索引块

    在索引数据块中流出一个存放下一个索引数据块的指针，当这个索引数据块的信息用完了，就可以找到下一个索引数据块的信息。但也存在之前链表的问题，指针损坏会导致后面的数据错误

    多级索引块

    索引+索引，通过一个索引块来放多个索引数据块

早期Unix的fs

根据文件所需的数据块数目阈值，通过不同方式存放。
文件头则可能包含 10、11、12、13个指针

空闲空间管理

要存一个数据块，则放在磁盘上的哪个位置？

空闲表法

为所有空间建立一张表，内容包括空闲去的第一个块号和空闲区的块的个数（连续分配方式）

这种方法仅当有少量的空闲区时才有较好的效果。因为，如果存储空间中有着大量的小的空闲区，则空闲表变得很大，这样查询效率会很低。另外，这种分配技术适用于建立连续文件。

空闲链表法

空闲块中有指针指向下一个空闲块。只需要在主存中保存一个指针，指向第一个空闲块

简单，但不能随机访问，效率低，IO多，不适合大型文件系统

位图法

利用二进制的一位表示磁盘中的一个盘块情况，所有的盘块都有一个二进制位与之对应。0表示空闲，1表示分配

Linux fs采用

fs结构

数据块的位图是放在磁盘块里的，假设是放在一个块里，一个块 4K，每位表示一个数据块，共可以表示 4 * 1024 * 8 = 2^15 个空闲块，由于 1 个数据块是 4K 大小，那么最大可以表示的空间为 2^15 * 4 * 1024 = 2^27 个 byte，也就是 128M。㘝通过一个块的位图+一系列的块最大也就表示128M的文件，如何解决这个问题

块组

N个块组 --> N大的文件

Linux Ext2:

  • 引导块：系统启动时来引导
  • 超级块：文件的重要信息，inode总个数、块总个数
  • 块组描述符：文件系统中各个块组的状态（块组中空闲快和inode的数目）
  • 数据位图+inode位图：表示对应的数据块或inode是否空闲
  • inode列表：包含了子块组中的所有inode，inode用于保存文件系统中与各个文件和目录相关的数据
  • 数据块…

目录的存储方式

通常，第一项是「.」，表示当前目录，第二项是「..」，表示上一级目录，接下来就是一项一项的文件名和 inode。
可以用vim打开，自测后发现每一行都是其下文件的名字（似乎只读）

保存目录的格式为哈希表 --> 提高查找效率，但需要避免哈希冲突

软链接硬链接

硬链接

多个目录项中的索引节点指向同一个文件，也就是指向同一个inode。由于inode无法跨越fs，每个fs都有自己的inode的数据结构和列表，所以hard link不能cross fs.由于多个目录项都指向一个inode，所以只有删除文件的所有硬链接和源文件时，系统才会彻底删除该文件？？？？文件在磁盘中只有一次复制，磁盘空间占用小
不能对文件夹进行硬链接，目的对象不能是已经存在的文件

源就是我、我就是源 ln 123.txt 1234.txt

软链接

相当于重新创建了一个文件，有独立的inode，但文件的内容却是另一个文件的路径。可以跨fs。删除源文件后就变成了死链接，不能使用。重新创建该目录的文件后，能够重新使用。目的对象不能是已经存在的文件

源就是我、我就是源 ln -s 123.txt 1234.txt

这里123.txt是源文件、12345.txt是硬链接、1234.txt是软连接

  1. 删除123.txt,软链接无法使用，硬链接照常用

文件IO

  • 缓冲与非缓冲IO

    • 缓冲：利用的是标准库的缓存实现文件的加速访问，而标准库再通过系统调用访问文件
      • 例如printf的换行输出
    • 非缓冲：直接用系统调用访问文件而不经过标准库缓存

  • 直接与非直接IO（内核：我不干扰）

    • 直接：不会发生内核缓存和用户程序之间数据复制，而是直接经过fs访问磁盘
    • 非直接：读操作时，数据从内核缓存中拷贝给用户程序，写操作时，数据从用户程序拷贝给内核缓存，再由内核决定什么时候写入数据到磁盘

  • 阻塞与非阻塞IO 同步与非同步IO

    • 阻塞：用户执行read，线程阻塞，等待内核数据准备好后，并把数据从内核缓冲区拷贝到应用程序的缓冲区后，read才会返回（等待：内核数据准备好 + 数据从内核态拷贝到用户态）

    • 非阻塞：read在数据未准备好的情况下立即返回，能继续执行，此时应用程序不断轮训内核，知道数据准备好，内核将数据拷贝到应用程序缓冲区，read调用能获得结果

      解决傻乎乎的轮训方式： IO多路复用 – 类似通知

      实际上，无论是阻塞 I/O、非阻塞 I/O，还是基于非阻塞 I/O 的多路复用都是同步调用。因为它们在 read 调用时，内核将数据从内核空间拷贝到应用程序空间，过程都是需要等待的，也就是说这个过程是同步的，如果内核实现的拷贝效率不高，read 调用就会在这个同步过程中等待比较长的时间。

    • 异步IO：发起aio_read后 立即返回，内核自动将数据进行拷贝，拷贝过程异步进行

总结一下：IO分为两个过程

  1. 数据准备
  2. 数据从内核空间拷贝到用户进程缓冲区

阻塞IO阻塞在1\2，非阻塞IO阻塞在2 —都是同步IO
异步IO 1、2都不会阻塞

例子

阻塞 I/O 好比，你去饭堂吃饭，但是饭堂的菜还没做好，然后你就一直在那里等啊等，等了好长一段时间终于等到饭堂阿姨把菜端了出来（数据准备的过程），但是你还得继续等阿姨把菜（内核空间）打到你的饭盒里（用户空间），经历完这两个过程，你才可以离开。

非阻塞 I/O 好比，你去了饭堂，问阿姨菜做好了没有，阿姨告诉你没，你就离开了，过几十分钟，你又来饭堂问阿姨，阿姨说做好了，于是阿姨帮你把菜打到你的饭盒里，这个过程你是得等待的。

基于非阻塞的 I/O 多路复用好比，你去饭堂吃饭，发现有一排窗口，饭堂阿姨告诉你这些窗口都还没做好菜，等做好了再通知你，于是等啊等（select 调用中），过了一会阿姨通知你菜做好了，但是不知道哪个窗口的菜做好了，你自己看吧。于是你只能一个一个窗口去确认，后面发现 5 号窗口菜做好了，于是你让 5 号窗口的阿姨帮你打菜到饭盒里，这个打菜的过程你是要等待的，虽然时间不长。打完菜后，你自然就可以离开了。

异步 I/O 好比，你让饭堂阿姨将菜做好并把菜打到饭盒里后，把饭盒送到你面前，整个过程你都不需要任何等待。
